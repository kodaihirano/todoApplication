(重要)MessageWindowのAPI再変更¶

ウィンドウの表示もメッセージ送りもメッセージ末尾での非表示もMessageWindow.next()として共通化しました。
(理由:末尾での非表示とその次の表示が同一フレーム内で実行されるのを防ぐにはこれしか思いつかなかった)

Scene.messageWindow.messages = ["message1", "message2"];
Scene.messageWindow.next(); // true, "message1"が書かれたウィンドウを表示
Scene.messageWindow.next(); // true, 文字列を"message2"に変更
Scene.messageWindow.next(); // false, ウィンドウを非表示

ウィンドウが消えるタイミングを取得するには返り値を使うと良さげです。

Scene.tlのダメな例¶

// EventScene.js line.16 in Initialize()
this.tl.action(()=>{});
for(const ev of this.events.event)this.parseEvent(ev);

// line.55 in parseEvent(ev)
case "map":

    this.initMap(ev[1], ev[2]);
  });
  break;

結果: EventScene中で実際にマップ切り替わる際、前後の処理によっては毎フレームinitMap()を繰り返したまま無限ループ
は？？？？？？？？？？？？？？？？？

(※上記map処理以外もthen()を使用すると多くが同じ挙動になる。仕様なのかバグなのかは不明)

Scene.tlの正解(仮)¶

// line.55 in parseEvent(ev)
case "map":
  this.tl.action({
    onactionstart(){this.tl.next();},
    onactionend(){
      this.initMap(ev[1], ev[2]);
    }
  });
  break;

結果: ちゃんと一度のみinitMap()が実行され、this.tlの次の処理へ進む。

フレームによるの線形変化について¶

フェードイン・フェードアウト等でフレーム毎に値を加算・減算する場合、変化値を2の整数乗にしないとバグる場合がある。
Entity.opacityの場合は 0.25, 0.125, 0.0625など。

enchant.jsのクラスと外部クラスとの連携(恐らくバグ)¶

グローバル変数gemaDataはES6記法のclass GameDataで定義されている。
> gameData = new GameData();

<失敗例>

// MapScene.js Line.63 
// for(const(int) i = ...) for(const(int) j = ...)

if(!gameData.box[this.mapname] || !gameData.opened(this.mapname, j, i)){ // always judged to be true.
  this.stage.addChild(new Box(this, obj[1], obj[2], j, i));
}

gameData.opened(this.mapname, j, i)が必ずfalseを返すらしい。
が、Chromeのコンソールから
> gameData.opened("samplemap0", 0, 5);
を叩くとちゃんとtrueが帰ってくる。gameData.box.samplemap0にもちゃんと[0,5]が格納されている。

<結論>

if(!gameData.box[this.mapname] || !gameData.opened(this.mapname, Number(j), Number(i))){
  this.stage.addChild(new Box(this, obj[1], obj[2], j, i));
}

納得いかない。

scene.removechild(this)の罠¶

子ノードからthis.scene.removechild(this)するとthis.sceneがnullになるっぽい。
なるべくしない方が良い。(代わりにイベントを消去する等で対応)

game.currentSceneの注意点¶

Sceneの子クラスはinitialize()が完了して初めてSceneスタックに積まれるらしく、initialize()の途中でgame.currentSceneを取得するとその手前のSceneになる。
initialize()内部で呼び出されるメソッド(コンストラクタ)に呼び出し元Sceneを教えたい場合は

// in FooScene.initialize(), where FooScene extends enchant.Scene
this.addChild(new Bar(this));

のように引数としてthisを渡すか、

// in FooScene.initialize(), where FooScene extends enchant.Scene
nextScene = this; // global variable
this.addChild(new Bar());

// in Bar.initialize()
this.scene = nextScene;

次のSceneとして仮置きするようにする。

スキル強化¶

Wandboxを使用する方針。日本語ドキュメントは偉大。
以下、APIをなんとなく触ってみて得た知見。

  • コンパイルするソースコードは単一ファイルのみだが入力ファイルは複数対応
    • なんとC/C++の場合はヘッダファイル(.h)をincludeすれば普通に分割コンパイルできる
  • 実行時オプションは使えない、というかスクリプト言語用の物でC言語には関係ないものらしい
    • 常に"$ gcc code [compiler-opton-raw]"→"$ ./a.out"が実行されるっぽい
    • 従って標準入力のリダイレクト("$ ./a.out < foo.txt"みたいなの)は不可

Wandboxを参考にDockerで環境構築中。

  • お披露目のとき、Dockerどうすんの？？？(計算機室のMacには入ってない)
    • ngrokとか？
    • 一旦Dockerを無視してローカルでgcc動かすのもアリ

  • Play!のモデルでアカウント管理
  • 「名前」「パスワード」「レベル・経験値」「所持金」など、プレイヤーと一対一関係のデータはアカウントと紐付けすればよい

    • ショップの在庫とか実装すると更に悲惨

解決案¶

以下のようなプレイヤーデータをmodelとは別に作る

{
  "playerID": 1,
  "subEvent":{
    "map5": [1],
  },
  "box":{
    "map0": [0,1],
    "map1": [1]
  }
}

この場合、プレイヤー1はmap5のサブイベントID1のサブイベントをこなし、map0の宝箱ID0と宝箱ID1の宝箱、map1の宝箱ID1の宝箱を開いたことになる。

  • アップデートの際にIDがズレてセーブデータの互換性が壊れるかもしれない
  • 店のデータの記述が悲惨になる
